如何正确升级 sqlite3 (CentOS)

2020-08-06
web
| 阅读

文章目录

  • 为 Django 升级 sqlite3
  • 为 uwsgi 升级原有的 sqlite3 动态链接库

为 Django 升级 sqlite3

在配置 Django 开发环境时, 如果采用默认的 sqlite3 作为数据库, 通常都会碰到 sqlite3 版本过低的运行报错, 像这样

在使用 Django 框架进行服务搭建的时候, 如果没有升级系统自带的 sqlite3 版本, name多数都会碰到类似下面这样的报错.

django.core.exceptions.ImproperlyConfigured: SQLite 3.8.3 or later is required (found 3.7.17).

根据这个报错可以在搜索引擎中找到一大堆关联的解决, 但是主要有两种方式

  1. 升级 sqlite3
  2. 降级django (如, 2.1.8)
  3. 更换后端数据库 (如, MySQL)

我觉得第三种方式, 更换数据库到 MySQL 也是个可以考虑的选择, 但是很多用 sqlite3 的项目非常的轻量, 大多是用不上 mysql 的, 至于降级 Django 就不推荐了, 那我们就看一下升级 sqlite3 的处理步骤吧:

  1. 官网获取 sqlite3 安装包
  2. 解压并编译安装 sqlite3
  3. 备份系统原有的 sqlite3, 替换新的 sqlite3 启动链接(/usr/bin/sqlite3).

流程基本是这样的

# 创建一个工作目录(可选)
mkdir sqlite3_upgrade
cd sqlite3_upgrade

# 下载源码包(大多发行版升级 sqlite3 都使用源码安装)
wget https://www.sqlite.org/2020/sqlite-autoconf-3320300.tar.gz
# 或者 wget https://docs.iterdaily.com/sqlite-autoconf-3320300.tar.gz

# 解压
tar -xf sqlite-autoconf-3320300.tar.gz

# 进入解压目录,编译安装
cd sqlite-autoconf-3320300
./configure --prefix=/usr/local
make && make install

# 替换系统低版本 sqlite3 启动文件
mv /usr/bin/sqlite3  /usr/bin/sqlite3_old
ln -s /usr/local/bin/sqlite3   /usr/bin/sqlite3
# 添加新版 sqlite3 动态链接库配置文件, 刷新动态链接库缓存
echo "/usr/local/lib" > /etc/ld.so.conf.d/sqlite3.conf
ldconfig

# 如果添加的 library 不在 /lib 或 /usr/lib 下, 但是却没有权限操作写 /etc/ld.so.conf 文件的话, 
# 这时就需要往 export 里写一个全局变量 LD_LIBRARY_PATH, 就可以了, 
# 最后将这个 export 命令写到 bashrc 等配置文件中.
export LD_LIBRARY_PATH="/usr/local/lib"

# 查看新的 sqlite3 版本是否生效
sqlite3 -version
3.32.3
python -c "import sqlite3;print(sqlite3.sqlite_version)"
3.32.3

至此, 新版本的 sqlite3 就应该已经能在 Django中正常使用了(而旧版本的依然保留下来了).

为 uwsgi 升级原有的 sqlite3 动态链接库

升级之后虚拟环境中 Django 配置使用也没有问题, 但是当开始 uwsgi 测试部署的时候, 这个 sqlite3 版本问题就又诡异的出现了.

uwsgi提示：SQLite 3.8.3 or later is required (found 3.7.17)

显然, 已经升级了的 sqlite3 这次并没有被 Django 虚拟环境所加载, 而是加载了升级之前系统默认的旧版本.

排查虚拟环境中的环境变量应该也是一个解决思路, 但是这样每个虚拟环境都要配置一遍, 所以就不采用这种方法了.

现在我们的解决思路是, 找到旧版本的动态链接库位置, 然后备份之后替换掉成新的就可以了 .

步骤大概是这样的

# ldconfig -v 
# 输出太长就不贴了

# 或者简单过滤一下看看有几条记录
# ldconfig -v |grep sqlite
ldconfig: Can't stat /libx32: No such file or directory
ldconfig: Path `/usr/lib' given more than once
ldconfig: Path `/usr/lib64' given more than once
ldconfig: Can't stat /usr/libx32: No such file or directory
        libsqlite3.so.0 -> libsqlite3.so.0.8.6
        libsqlite3.so.0 -> libsqlite3.so.0.8.6

# 上面的报错可以忽略, 但是有两个 sqlite3 的动态链接库配置, 
# 一个在 /usr/lib64/ 这个是默认的, 另一个在 /usr/local/lib/ 这个是我们新编译安装的, 
# 现在将旧版本备份一下, 更新一下链接文件就可以了.

# rpm -ql sqlite
/usr/bin/sqlite3
/usr/lib64/libsqlite3.so.0
/usr/lib64/libsqlite3.so.0.8.6
/usr/share/doc/sqlite-3.7.17
/usr/share/doc/sqlite-3.7.17/README
/usr/share/man/man1/sqlite3.1.gz

# ll /usr/lib64/libsqlite3.so.0
lrwxrwxrwx 1 root root     19 Jul 27 06:27 /usr/lib64/libsqlite3.so.0 -> libsqlite3.so.0.8.6

# 备份原有文件, 要注意文件名不要再使用 .so.0.0.8.6 后缀
mv /usr/lib64/libsqlite3.so.0.8.6 /usr/lib64/libsqlite3_so_0_8_6_old

# ls /usr/local/lib/ -l
total 18312
-rw-r--r-- 1 root root 13245026 Aug  5 16:05 libsqlite3.a
-rwxr-xr-x 1 root root      975 Aug  5 16:05 libsqlite3.la
lrwxrwxrwx 1 root root       19 Aug  5 16:05 libsqlite3.so -> libsqlite3.so.0.8.6
lrwxrwxrwx 1 root root       19 Aug  5 16:05 libsqlite3.so.0 -> libsqlite3.so.0.8.6
-rwxr-xr-x 2 root root  5498352 Aug  5 16:05 libsqlite3.so.0.8.6
drwxr-xr-x 2 root root       24 Aug  5 16:05 pkgconfig
drwxr-xr-x 3 root root       27 Aug  3 16:29 python3.6

# 替换旧文件
cp /usr/local/lib/libsqlite3.so.0.8.6 /usr/lib64/libsqlite3.so.0.8.6

# ldconfig #刷新一下库缓存

# ll /usr/lib64/libsqlite3*
lrwxrwxrwx  1 root root      19 Aug  6 15:48 /usr/lib64/libsqlite3.so.0 -> libsqlite3.so.0.8.6
-rwxr-xr-x  1 root root 5498352 Aug  6 15:36 /usr/lib64/libsqlite3.so.0.8.6
-rwxr-xr-x. 1 root root  753232 Nov 20  2015 /usr/lib64/libsqlite3_so_0_8_6_old

现在问题就解决了.

这个问题网上很多答案都没有给出解决, 其实是, 系统自带的 sqlite3 生成的动态链接库在 /usr/lib64/ 下面, 这也是系统默认的动态链接库存放的位置, uwsgi 直接到了这个默认位置去加载 sqlite3 也就直接使用了旧版本的, 所以我们把默认的链接库也更新一下就可以正常使用了.
